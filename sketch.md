# 目标

这个 skill 用于为中小需求生成一份轻量、可执行的实现计划。

适用场景：
- 需求不算特别大，但也不是一句话就能直接改
- 需要先看代码再决定怎么做
- 需要和用户对齐一些关键点
- 希望先产出 plan，再决定是否进入实现

这个 skill 的核心目标是输出一份聚焦"实现方案和改动点"的 `sketch.md`。

# 总原则

1. 当前处于规划模式，不是实现模式。
2. 在用户确认前，不要直接修改业务代码。
3. 可以进行只读探索。
4. 唯一允许创建或更新的产物是：
    - `specs/{feature_name}/sketch.md`
5. 如果工作区目录不存在，可以创建：
    - `specs/{feature_name}/`

# 何时使用

满足以下任一条件时使用：
- 需求涉及多个文件或模块
- 需要先理解现有实现再决定改法
- 用户明确要求先出 plan
- 存在关键不确定点，需要先澄清

以下情况通常不必使用：
- 极小改动
- 单文件、低风险、实现方式非常明确的修改

# 执行流程

## 1. 理解需求并生成工作区名称

先把用户需求整理成：
- 要做什么
- 不做什么
- 当前已知约束
- 还缺什么关键信息（初始待解问题列表）

如果需求存在**方向性歧义**（范围不明、多个方向差异大、验收标准缺失），在进入探索前先**向用户提问**，问题数量尽量少，只问方向性的。能通过读代码确认的不要问。

然后生成一个简短的 kebab-case 名称，作为 `feature_name`。

要求：
- 2 到 6 个词
- 小写
- 使用连字符分隔
- 贴近用户真实需求

示例：
- `add-user-export`
- `fix-loan-status-race`
- `refactor-risk-rule-eval`

工作区路径：
- `specs/{feature_name}/`

## 2. 问题驱动的探索循环

维护一个**待解问题列表**，从需求分析中提取初始问题，然后循环推进，直到没有影响方案正确性的未解问题为止。

**每轮开始前**：从待解问题列表中挑出本轮能通过读代码解答的问题，组织为探索目标，**启动只读探索**（不修改任何文件）。每轮的目标必须围绕当前具体问题现场撰写，不套固定模板。

**每轮结束后**：
- 将已解答的问题从列表中移除
- 将探索中新发现的问题加入列表
- 如果列表中还有只有用户才能回答的问题（产品决策、优先级、验收标准），**向用户提问**，将用户回答补充进列表
- 判断是否继续：列表中还有影响方案正确性的未解问题 → 继续；否则 → 退出循环

**终止条件**：满足以下全部条件时退出循环：
- 每个需要改动的文件已定位到具体路径（最好到符号/函数级）
- 没有"不确定用哪个方案"的分叉点悬而未决
- 没有依赖项尚未确认存在（接口、字段、配置项等）

---

提问原则（适用于向用户提问）：
- 只问高价值问题，问题数量尽量少
- 能通过读代码确认的，不要问用户
- 每个问题尽量提供选项，方便用户快速回答

---

**常见模式（启发式参考，不是强制阶段）**：
- 第 1 轮通常聚焦于定位结构：相关模块、调用链入口、主要数据结构
- 中间轮次深入具体文件：核实字段定义、方法签名、配置结构
- 最后一轮往往是验证草拟改动点的依赖：被调用的方法是否存在、基础设施状态是否符合预期、上下游调用者是否需要联动

以当前待解问题为准，不要机械套用轮次。

**sketch.md 中不得出现"待用户确认的问题"。**

## 3. 收敛方案并写 sketch.md

待解问题全部解决后，输出一份轻量 `sketch.md`，写入 `specs/{feature_name}/sketch.md`。

这份计划应突出：
- 目标
- 推荐实现方案
- 相关改动 & 步骤
- 验证方式

不要把它写成冗长 spec。
不要堆过多背景说明。

## 4. 结束于计划，不进入实现

写完 `sketch.md` 后：
- 简要总结推荐方案
- 标出已知的技术风险和假设
- 等待用户批准

用户批准后，**直接按 sketch.md 中的改动步骤逐项实现**，使用标准代码编辑工具完成。不要调用 SDD 的 tasks / implement 等其他 skill——sketch 是轻量路径，不走完整 SDD 流水线。

# sketch.md 写作要求

`sketch.md` 应尽量短，但要足够可执行。

必须包含：
1. 目标
2. 推荐实现方案
3. 相关改动 & 步骤
5. 验证方式

必须尽量引用真实文件路径。
如果能定位到符号或职责点更好，例如：
- `src/order/service.ts`
- `src/order/service.ts::createOrder`
- `backend/loan/handler.py (status transition logic)`

如果某个文件只是高概率会改，但还不完全确定，要明确写成"可能涉及"。

# sketch.md 示例

```markdown
## 目标
为导出报表功能增加"按部门过滤"选项。

## 推荐方案
在现有导出接口增加可选参数 `dept_id`，后端查询时拼入 WHERE 条件，前端下拉选择器复用已有的部门组件。

## 改动点
- `src/report/export_handler.py::export_report` — 新增 `dept_id` 参数，传入查询层
- `src/report/query.py::build_export_query` — 当 `dept_id` 非空时追加 `AND dept_id = ?`
- `src/frontend/ExportModal.vue` — 加部门下拉，绑定 `dept_id`；部门列表复用 `useDeptOptions()`（已有）
- `src/api/report.ts::exportReport` — 参数类型补 `deptId?: string`

## 验证方式
- 不传 `dept_id`：导出结果与现有行为一致
- 传入有效 `dept_id`：结果只含该部门数据
- 传入无权限的 `dept_id`：返回 403（沿用现有鉴权逻辑，待确认覆盖范围）
```

# 质量要求

好的计划应当：
- 能让另一个 agent 直接开始实现
- 明确改哪些文件、为什么改
- 说明主要方案，而不是泛泛而谈
- 把技术层面的不确定点和风险显式写出来
- 包含最基本的验证方式

差的计划通常表现为：
- 空泛
- 没有文件路径
- 只有"修改后端逻辑""补充前端逻辑"这种废话
- 把 spec、设计文档、任务拆解全塞进来，导致过重

# 输出风格

整体保持：
- 中文
- 简洁
- 工程化
- 面向实施

避免：
- 长篇背景铺垫
- 套话
- 过多"最佳实践"空话
